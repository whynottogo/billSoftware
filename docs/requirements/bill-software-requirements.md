# 记账软件需求文档

## 1. 文档信息

- 文档名称：记账软件需求文档
- 文档版本：v1.0
- 更新时间：2026-04-09
- 适用范围：PC 端记账软件 V1
- 技术栈约束：
  - 前端：Vue 3、Element Plus、Axios、Webpack、ECharts、Babel 6
  - 后端：Golang、Gin、XORM

## 2. 文档目标

本文件用于沉淀 V1 版本的正式需求，供产品设计、前后端开发、测试以及多 AI 协同开发时统一读取与执行。

本文件必须同时满足以下目标：

1. 先说明系统级规则，避免实现时出现权限、登录、路由、账号体系混乱。
2. 再说明模块级需求，让开发方理解每个模块的业务边界和页面行为。
3. 最后拆成可并行认领的任务卡，每条任务都带状态与负责 AI，避免多 AI 写串。

## 3. 全局术语

- 用户端：普通用户使用的 PC 端系统，访问前缀为 `/user`。
- 管理端：唯一管理员使用的 PC 端系统，访问前缀为 `/admin`。
- 用户账号：注册产生的普通用户账号，存储于用户体系。
- 管理员账号：独立于用户体系的固定管理员账号，不属于用户表，不允许登录用户端。
- 收入记录：用户新增的收入流水。
- 支出记录：用户新增的支出流水。
- 账单：按月或按年汇总后的统计结果。
- 资产账户：资产管家中的账户对象，用于维护资产、负债及余额变动记录。
- 弱联动：收入/支出流水不会自动改动资产账户余额，资产余额只能由资产管家单独维护。
- 家庭：多个用户组成的共享统计单元，用于查看家庭汇总数据。

## 4. 协同开发字段规范

所有需求任务必须包含以下字段：

| 字段名 | 含义 | 示例 |
| --- | --- | --- |
| `req_id` | 唯一需求编号 | `USER-AUTH-001` |
| `side` | 所属端 | `user` / `admin` / `system` |
| `module` | 所属模块 | `登录注册` |
| `status` | 当前状态 | `未开始` |
| `owner_ai` | 当前负责 AI | `待分配` |
| `dependencies` | 依赖任务编号数组 | `["SYS-AUTH-001"]` |
| `detail` | 详细需求说明 | 文本 |
| `acceptance_criteria` | 验收标准 | 编号列表 |
| `notes` | 备注或约束 | 文本 |

状态字段统一使用以下枚举：

- `未开始`
- `开发中`
- `联调中`
- `测试中`
- `已完成`
- `已阻塞`

`owner_ai` 默认值统一为 `待分配`，后续按实际认领人替换，例如 `AI-前端-01`、`AI-后端-02`。

### 4.1 状态口径补充（2026-04-09）

1. `已完成` 仅用于当前实现已经满足该 `req_id` 的 Acceptance Criteria。
2. 若需求 `Notes` 中明确说明 `V1 使用 mock 数据`、`不要求接入真实后端接口` 或 `仅实现原型级页面`，则 mock / 原型实现可作为 `已完成` 依据。
3. 若仅完成 Figma / Make 到 Vue 的页面翻写、本地 mock 或 `localStorage` 演示，且 `Notes` 未明确允许 mock，则状态必须保持为 `开发中`。
4. 若前后端都已有部分实现，但前端仍未完成真实接口接线或仅完成局部联调，则状态应标记为 `联调中`。

## 6. 系统规则

### 6.1 系统拆分与路由规则

1. 本项目只有一个前端工程，但必须按路由前缀拆成两个系统。
2. 用户端所有页面必须放在 `/user` 前缀下。
3. 管理端所有页面必须放在 `/admin` 前缀下。
4. 推荐的 V1 页面路由如下：

| side | 路由 | 页面 |
| --- | --- | --- |
| user | `/user/login` | 用户登录 |
| user | `/user/register` | 用户注册 |
| user | `/user/ledger` | 当月收支首页 |
| user | `/user/bills/month` | 月账单列表 |
| user | `/user/bills/month/:month` | 月账单详情 |
| user | `/user/bills/year` | 年账单 |
| user | `/user/budget/month` | 月预算 |
| user | `/user/budget/year` | 年预算 |
| user | `/user/assets` | 资产管家 |
| user | `/user/assets/:accountId` | 账户详情 |
| user | `/user/charts/expense` | 支出图表 |
| user | `/user/charts/income` | 收入图表 |
| user | `/user/profile` | 个人信息 |
| user | `/user/profile/password` | 修改密码 |
| user | `/user/families` | 家庭列表 |
| user | `/user/families/:familyId` | 家庭详情 |
| admin | `/admin/login` | 管理员登录 |
| admin | `/admin/dashboard` | 数据概览 |
| admin | `/admin/users` | 用户列表 |
| admin | `/admin/users/:userId` | 用户详情 |
| admin | `/admin/approvals` | 待审批用户 |
| admin | `/admin/families` | 家庭管理 |

### 6.2 账号体系规则

1. 用户账号和管理员账号必须是两套完全独立的账号体系。
2. 用户账号来自注册流程，存储于用户表。
3. 管理员账号不在用户表中，不属于用户管理范围。
4. 管理端只有 1 个固定管理员账号。
5. 不提供管理员列表。
6. 不提供管理员新增、删除、编辑、禁用功能。
7. 管理员账号来源固定为后端环境配置，不通过界面维护。
8. 建议的环境变量名如下：
   - `BILL_ADMIN_USERNAME`
   - `BILL_ADMIN_PASSWORD_HASH`
9. 用户登录接口只能校验用户体系，绝不能回落到管理员配置。
10. 管理员登录接口只能校验管理员环境配置，绝不能查询用户表。
11. 管理员账号不能登录用户端。
12. 普通用户账号不能登录管理端。

### 6.3 鉴权与会话规则

1. 用户端和管理端必须分别拥有独立的认证对象、独立的登录入口、独立的会话校验逻辑。
2. 用户端会话和管理端会话不得共用前端存储 key。
3. 用户端会话和管理端会话不得共用服务端认证中间件。
4. 用户端必须实现单会话规则：
   - 同一用户账号在 PC 端只能同时保持一处登录状态。
   - 当同一账号在新设备或新浏览器登录成功后，旧登录会话立即失效。
   - 旧登录会话访问接口时应返回未登录或会话失效。
   - 前端应提示“账号已在其他地方登录，请重新登录”。
5. 管理端默认允许管理员正常多处登录，不做互踢限制。
6. 用户端和管理端的互踢逻辑互不影响。

### 6.4 用户状态与注册规则

1. 用户注册后默认状态为 `禁用`。
2. 只有管理员在管理端启用该用户后，该用户才允许登录用户端。
3. 用户状态 V1 只保留 `启用` 和 `禁用` 两种。
4. 不单独设计“待审批”状态字段。
5. “管理员启用用户”即视为审批通过。
6. 禁用用户即使密码正确也不能登录。

### 6.5 数据隔离规则

1. 用户端所有业务数据默认仅允许本人查看和操作。
2. 需要按用户隔离的数据至少包括：
   - 收入记录
   - 支出记录
   - 收支分类
   - 预算
   - 资产账户
   - 账户余额变动记录
   - 图表数据
   - 个人资料
3. 家庭功能属于例外共享场景，但共享范围仅限家庭汇总数据和成员占比数据。
4. 管理端仅允许管理员查看用户列表和用户账单数据。
5. 管理员不应出现在用户管理对象中。

### 6.6 上传与图片规则

1. 每条收入记录最多支持 1 张图片。
2. 每条支出记录最多支持 1 张图片。
3. 图片为可选字段，不上传也可保存记录。
4. 图片在前端需支持预览。
5. 个人头像上传后需保留：
   - 一张原始头像
   - 一张压缩头像
6. 头像若超过 2M，需要压缩到不高于 2M。
7. 头像原始图与压缩图沿用原始需求，默认存数据库。
8. 收支记录图片在 V1 只要求支持上传、展示、预览，不强制规定最终存储实现。

### 6.7 账单与资产关系

1. 账单与资产账户采用弱联动设计。
2. 新增收入记录不会自动增加资产账户余额。
3. 新增支出记录不会自动减少资产账户余额。
4. 账户余额只能通过资产管家中的余额调整、增加余额、减少余额功能维护。
5. 账单模块负责统计收入与支出。
6. 资产模块负责统计资产、负债、净资产。

### 6.8 V1 明确不做

1. 不做管理员管理模块。
2. 不做角色管理模块。
3. 不做权限配置中心。
4. 不做系统设置模块。
5. 不做审计日志模块。
6. 不做账单与资产自动联动。
7. 不做家庭成员审批、移除、共管规则。
8. 不做收支记录多图上传。

## 7. 模块需求说明

### 7.1 登录注册

用户端登录注册模块必须支持：

1. 用户注册。
2. 用户登录。
3. 用户退出登录。
4. 用户被互踢后的失效提示。

业务规则：

1. 注册字段包括用户名、昵称、密码、手机号、邮箱。
2. 用户名唯一。
3. 手机号唯一。
4. 用户端支持“用户名或手机号 + 密码”登录。
5. 注册成功后不能立即登录，必须先被管理员启用。

### 7.2 当月收支

该模块是用户端默认核心页面，需承接当月查看与记账。

1. 页面按天展示指定月份的收支记录。
2. 每天显示当日总收入与总支出。
3. 每条记录显示金额、类型、备注、日期、图片。
4. 允许新增支出记录。
5. 允许新增收入记录。
6. 允许管理收入分类。
7. 允许管理支出分类。

修正后的默认分类如下：

- 默认支出分类：
  - 早午晚餐、餐饮、购物、日用、奶茶、交通、蔬菜、水果、零食、运动、娱乐、通讯、服饰、美容、住房、居家、孩子、长辈、社交、旅行、烟酒、数码、汽车、医疗、学习、宠物、礼物、办公、维修、捐赠、彩票、亲友、快递
- 默认收入分类：
  - 工资、兼职、理财、礼金、其他

说明：

1. 原始记录中的收入和支出默认分类存在写反问题，本文件已按正确语义修正。
2. 收入分类和支出分类都必须按用户独立维护，不共享。

### 7.3 账单

账单模块包含：

1. 月账单列表。
2. 月账单详情。
3. 年账单。

月账单列表：

1. 展示本年结余、年收入、年支出。
2. 展示月份列表。
3. 每个月显示月收入、月支出、月结余。
4. 点击某个月进入月账单详情。

月账单详情：

1. 展示本月结余、上月结余。
2. 展示本月收入、本月支出。
3. 展示支出分类饼图。
4. 展示支出排行。
5. 展示支出趋势折线图。
6. 展示与前 5 个月对比的月支出柱状图。
7. 展示与前 5 个月对比的月收入柱状图。
8. 展示记账成就：连续打卡天数、记账总天数、记账总笔数。

年账单：

1. 展示总结余、总收入、总支出。
2. 展示年份列表。
3. 每年显示年收入、年支出、年结余。

### 7.4 预算

预算模块包括月预算和年预算。

月预算：

1. 只能设置和查看当月预算。
2. 展示本月预算、本月支出、本月剩余预算。
3. 支持按支出类型设置细分预算。
4. 支持查看各支出类型剩余预算。

年预算：

1. 只能设置和查看当年预算。
2. 展示本年预算、本年支出、本年剩余预算。
3. 支持按支出类型设置细分预算。
4. 支持查看各支出类型剩余预算。

说明：

1. 原始记录中“年预算”的文案误写为“本月支出、本月剩余预算”，本文件已修正为“本年支出、本年剩余预算”。

### 7.5 资产管家

资产管家模块包含：

1. 资产总览。
2. 账户列表。
3. 账户详情。

资产总览：

1. 顶部显示净资产、资产、负债。
2. 计算公式为：资产 - 负债 = 净资产。

账户列表：

1. 按账户分类展示账户列表。
2. 每个分类展示该分类总资产或总负债。
3. 每个账户展示该账户当前余额。
4. 支持新增账户。

账户类型包括：

- 现金
- 储蓄卡
- 信用卡
- 虚拟账户
- 投资账户
- 负债
- 债权
- 自定义资产

账户字段规则：

1. 新增账户时必须填写名称、备注、余额。
2. 银行卡和信用卡必须额外填写卡号。
3. 虚拟账户需要区分微信或支付宝。
4. 信用卡和负债类账户视为负债。
5. 其他账户类型视为资产。

账户详情：

1. 支持调整余额。
2. 支持增加余额。
3. 支持减少余额。
4. 支持查看余额变动记录列表。
5. 记录列表按月切换查看。
6. 设置项允许修改备注、名称、余额。
7. 若设置项修改余额，也必须生成一条调整余额记录。
8. 账户类型创建后不可修改。

### 7.6 图表

支出图表：

1. 支持按月查看支出统计。
2. 支持按年查看支出统计。
3. 月视图使用按天折线图。
4. 年视图使用按月折线图。
5. 展示支出前 10 类型排行榜，包含类型、金额、占比。

收入图表：

1. 支持按年查看收入折线图。
2. 展示收入前 10 类型排行榜，包含类型、金额、占比。

### 7.7 个人信息

查看与修改个人信息：

1. 账户不可修改。
2. 手机号不可修改。
3. 昵称可修改。
4. 头像可修改。
5. 邮箱可修改。

密码修改：

1. 字段包括原密码、新密码、重复新密码。
2. 原密码校验通过后才允许修改。
3. 新密码与重复新密码必须一致。

### 7.8 家庭功能

家庭功能属于用户增强功能。

1. 用户可以查看家庭列表。
2. 家庭列表中显示家庭名称、创建人。
3. 用户可以创建家庭。
4. 创建家庭时需要填写家庭名称和家庭编号。
5. 每个家庭必须有独立 id 字符串。
6. 用户可以通过家庭 id 加入家庭。
7. 用户可以通过邀请链接加入家庭。
8. 用户可以主动退出某个家庭。
9. 不设计家庭审批。
10. 不设计家庭移除成员。
11. 不设计家庭共管权限。

家庭详情：

1. 支持查看指定月份的月收入、月支出、月结余。
2. 支持查看指定年份的年收入、年支出、年结余。
3. 点击月收入、月支出、年收入、年支出时，显示家庭成员占比饼图。

### 7.9 管理端

管理端只包含最小可用范围。

管理员登录：

1. 只有固定管理员账号可以登录。
2. 管理员账号由环境配置提供。
3. 管理员账号不能进入用户端。

用户管理：

1. 支持查看用户列表。
2. 支持启用用户。
3. 支持禁用用户。
4. 支持查看用户详情。
5. 用户详情页包含月和年两个 tab。
6. 可查看用户某个月或某年的收入、支出、总结余。

后台门户：

1. 支持查看数据概览页。
2. 支持查看待审批用户页。
3. 支持查看家庭管理页。
4. 数据概览页展示用户增长、账单趋势和最近注册用户。
5. 待审批用户页展示待审批清单和批量操作入口。
6. 家庭管理页展示家庭统计和家庭列表。

后台推荐路由：

1. `/admin/dashboard` 数据概览。
2. `/admin/users` 用户管理。
3. `/admin/users/:userId` 用户详情。

## 9. 验收检查清单

在进入编码前、联调前、测试前，均需按以下清单检查：

1. 每个任务是否都具备 `req_id`、`status`、`owner_ai`、`dependencies`。
2. 用户端与管理端是否已经按两套账号体系设计。
3. 用户端单会话规则是否只作用于用户端。
4. 禁用用户是否无法登录。
5. 管理员账号是否不出现在用户管理中。
6. 账单与资产是否仍保持弱联动。
7. 收支分类默认值是否已按本文件修正后的版本实现。
8. 年预算文案是否已按“本年预算、本年支出、本年剩余预算”实现。
9. 家庭功能是否没有引入审批、移除、共管等未定义能力。
10. 图片能力是否限制为单条收支记录最多 1 张图。

## 10. 后续协同建议

建议按以下顺序认领任务：

1. 先完成系统任务。
2. 再完成登录注册与鉴权。
3. 再完成当月收支与分类管理。
4. 再完成账单、预算、图表。
5. 再完成资产管家。
6. 最后完成家庭功能和管理端查看能力。

建议的并行边界：

1. 用户登录注册和管理员登录可并行。
2. 当月收支与资产管家可并行，但双方必须遵守弱联动约束。
3. 图表与账单可并行，但统计口径必须由同一后端汇总逻辑统一输出。
